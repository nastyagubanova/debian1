        Часть 1
cap_net_raw - capability (сырые сетевые пакеты)
e = effective (активна сейчас)
p = permitted (разрешена для процесса)
в системе обнаружены 3 файла:
/usr/bin/dumpcap
/usr/bin/ping
/usr/lib/x86_64-linux...

CapPrm (Permitted) - разрешённый набор, который процесс может использовать
CapEff (Effective) - реально активные сейчас
CapBnd (Bounding set) - максимальный набор, который процесс вообще может получить
В выводе у меня нет на данный момент активных capabilities.

setcap безопаснее sudo, потому что даёт только нужную capability без полного root-доступа

Флаги:
e - capability активна сейчас (effective)
i - наследуется дочерним процессом (inheretable)
p - разрешена в наборе permitted

        Часть 2
В режиме enforce - 24 профиля, в режиме complain - 23 профиля.
Программа man разрешено читать справочные файлы и запускать ограниченные системные операции.
Используется декларативный синтаксис AppArmor с правилами доступа вида allow/read/man path и профилем на основе путей.

Без профиля скрипт свободно выполнил все действия: прочитал оба файла и смог записывать как в /tmp так и в /etc буз ограничений
| Действия скрипта | Без профиля | complain | enforce |
| Читать /tmp/pr07-allowed.txt | OK | OK | OK |
| Читать /etc/shadow | OK | LOG + OK | DENIED |
| Писать /tmp/pr07-output.txt | OK | OK | OK |
| Писать /etc/ | OK | LOG + OK | DENIED |

В логах же разницы не было и ничего не записывалось не смотря на првильное следование указаниям. В причине разобраться не смогла. Строк DENIED не находилось не смотря на блокировку доступа.

        Часть 3.
        
| Ресурс | Хост | Контейнер |
| Количество проессов | ~N | 1 |
| Сетевые интерфейсы | eth0, lo, docker0 | lo, ens33, docker0 |
| Корневая система | полная | Собственная файловая система образа убунту |
| /etc/shadow хоста | доступен | недоступен |

У контейнера отсутствуют многие capabilities: CAP_SYS_ADMIN, CAP_SYS_MODULE, CAP_SYS_TIME, CAP_NET_ADMIN, CAP_SYS_BOT, CAP_AUDIT_CONTROL

Монтирование конкретной папки безопаснее, потому что контейнер получайет доступ только к нужному каталогу, а не почти полный контроль над хостовой системой.

Запуск не от рут важен, потому что снижает последствия возможного взлома: процесс получает только права обычного пользователя и не может выполнять административные действия внутри контейнера и за его пределами.

        Итоговое задание
Nginx выданы только необходимые: NET_BIND_SERVICE для прослушивания порта 80, SHOWN, DAC_OVERRIDE, SETUID и SETGID для работы с файлами и смены пользователя. Это соответствует принципу минимальных привилегий.
